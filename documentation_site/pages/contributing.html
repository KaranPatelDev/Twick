Contributing to Twick

Help make Twick better! We welcome contributions from developers of all skill levels.

Ways to Contribute

Report Bugs

Found a bug? Help us fix it by creating a detailed bug report with steps to reproduce the issue.

Suggest Features

Have an idea for a new feature? We'd love to hear it! Submit a feature request with detailed specifications.

Submit Code

Contribute code improvements, bug fixes, or new features. All pull requests are welcome!

Improve Documentation

Help improve our documentation by fixing typos, adding examples, or writing new guides.

Getting Started

1

Fork the Repository

Start by forking the Twick repository to your GitHub account:

# Clone your fork
git clone https://github.com/YOUR_USERNAME/twick.git
cd twick

# Add upstream remote
git remote add upstream https://github.com/original-username/twick.git
2

Set Up Development Environment

Follow our development setup guide:

# Create virtual environment
python -m venv venv
source venv/bin/activate  # Linux/macOS
venv\Scripts\activate     # Windows

# Install dependencies
pip install -r requirements.txt
pip install -r requirements-dev.txt

# Set up pre-commit hooks
pre-commit install
3

Create a Branch

Create a new branch for your feature or bug fix:

# Create and switch to new branch
git checkout -b feature/your-feature-name
# or
git checkout -b bugfix/issue-number-description
4

Make Your Changes

Implement your feature or bug fix following our coding standards.

5

Test Your Changes

# Run tests
python manage.py test

# Run code quality checks
flake8 .
black . --check
isort . --check-only
6

Submit Pull Request

Push your changes and create a pull request with a clear description.

Development Guidelines

Code Style

Python Code
  • Follow PEP 8 style guidelines
  • Use Black for code formatting
  • Use isort for import sorting
  • Maximum line length: 88 characters
  • Use type hints when possible
Django Conventions
  • Follow Django naming conventions
  • Use Django's built-in features when available
  • Write proper model relationships
  • Use Django forms for data validation
  • Follow DRY (Don't Repeat Yourself) principles

Frontend Guidelines

HTML/CSS
  • Use semantic HTML elements
  • Follow Bootstrap conventions
  • Make responsive designs
  • Use ARIA labels for accessibility
  • Optimize for mobile-first
JavaScript
  • Use modern ES6+ syntax
  • Follow consistent naming conventions
  • Add proper error handling
  • Write modular, reusable code
  • Document complex functions

Testing Guidelines

Unit Tests

Write unit tests for all new functionality:

# Example test
from django.test import TestCase
from django.contrib.auth.models import User
from tweet.models import Tweet

class TweetModelTest(TestCase):
    def setUp(self):
        self.user = User.objects.create_user(
            username='testuser',
            password='testpass123'
        )
    
    def test_tweet_creation(self):
        tweet = Tweet.objects.create(
            user=self.user,
            content='Test tweet'
        )
        self.assertEqual(tweet.content, 'Test tweet')
        self.assertEqual(tweet.user, self.user)

Integration Tests

Test API endpoints and user flows:

# Example API test
from rest_framework.test import APITestCase
from django.urls import reverse
from rest_framework import status

class TweetAPITest(APITestCase):
    def setUp(self):
        self.user = User.objects.create_user(
            username='testuser',
            password='testpass123'
        )
        self.client.force_authenticate(user=self.user)
    
    def test_create_tweet(self):
        url = reverse('tweet-list')
        data = {'content': 'Test tweet'}
        response = self.client.post(url, data)
        self.assertEqual(response.status_code, status.HTTP_201_CREATED)
Test Coverage

We aim for at least 80% test coverage. Run coverage reports with:

coverage run --source='.' manage.py test
coverage report
coverage html  # Generate HTML report

Commit Guidelines

We follow conventional commit format for clear and consistent commit messages:

Commit Format

type(scope): description

[optional body]

[optional footer(s)]
Commit Types
  • feat: New feature
  • fix: Bug fix
  • docs: Documentation changes
  • style: Code style changes
  • refactor: Code refactoring
  • test: Adding tests
  • chore: Maintenance tasks
Examples
feat(auth): add social login with Google

fix(tweet): resolve character limit validation

docs(api): update authentication documentation

style(frontend): format CSS files with prettier

test(models): add tests for Tweet model

Pull Request Guidelines

Before Submitting

PR Template

## Description
Brief description of the changes made.

## Type of Change
- [ ] Bug fix (non-breaking change that fixes an issue)
- [ ] New feature (non-breaking change that adds functionality)
- [ ] Breaking change (fix or feature that causes existing functionality to not work as expected)
- [ ] Documentation update

## Testing
- [ ] Unit tests pass
- [ ] Integration tests pass
- [ ] Manual testing completed

## Screenshots (if applicable)
Add screenshots to help explain your changes.

## Related Issues
Fixes #123
Closes #456
